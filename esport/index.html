E-Sport Club Management System

A robust microservices-based platform designed to streamline operations for e-sports organizations, from player management to match scheduling and performance analysis.

Project Highlights

Microservice Architecture

Deployed on AWS EC2, ensuring scalability, fault tolerance, and independent service management for ultimate reliability.

Role-Based Access Control

Secure JWT-based authentication protects over 20 API endpoints, granting specific permissions to Players, Coaches, and Admins.

Full CRUD Functionality

Comprehensive management of users, schedules, and VOD reviews, with a flexible MongoDB backend.

Methodology Deep Dive

The system's backend is a microservices architecture built with Node.js and Express.js, chosen for their performance and robust ecosystem. This design isolates key functionalities—such as user registration, coach/admin actions, and player data—into separate, scalable services deployed on AWS EC2 instances. This ensures high availability and simplifies maintenance. For data persistence, MongoDB was selected for its flexible, schema-less structure, which is ideal for evolving e-sports data like player stats and match schedules...

Each microservice is configured to run on a dedicated port to prevent conflicts and streamline service communication. An API Gateway (Port 4000) serves as the single entry point for all client requests, routing them to the appropriate service: Registration/Auth (5001), Coach/Admin (5002), or Player (5003). This structured port allocation enhances security and simplifies load balancing.

Security is paramount, with JSON Web Tokens (JWT) handling API authentication. Upon login, a secure token is generated, validating user identity and role for all subsequent requests without needing server-side sessions. Middleware like `body-parser` handles incoming JSON payloads, while `dotenv` manages environment variables, keeping sensitive credentials like database URIs and JWT secrets securely outside the source code.

API Request Flow (Sequence)

sequenceDiagram
    participant C as 🖥️ Client
    participant G as 🚪 API Gateway
    participant A as 🔐 Auth Service
    participant Coach as 👨‍💼 Coach Service
    participant P as 🎮 Player Service
    participant DB as 🗄️ MongoDB
    
    Note over C,DB: User Registration
    C->>G: POST /reg/player
    G->>A: Forward registration request
    A->>DB: Store user data
    DB-->>A: User created
    A-->>G: Registration success
    G-->>C: User registered
    
    Note over C,DB: Authentication
    C->>G: POST /auth/login
    G->>A: Forward login request
    A->>DB: Validate credentials
    DB-->>A: User found
    A-->>G: JWT Token
    G-->>C: Login successful + JWT
    
    Note over C,DB: VOD Assignment Flow
    C->>G: POST /coach/assignvod + JWT
    G->>Coach: Forward VOD assignment
    Coach->>DB: Create VOD assignment
    DB-->>Coach: Assignment created
    Coach->>P: Notify player service
    Coach-->>G: Assignment success
    G-->>C: VOD assigned
